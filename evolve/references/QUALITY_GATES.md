# QUALITY_GATES — `evolve`-specific Phase 4/5 enforcement

This file contains only what is **specific to `evolve`** — i.e. brownfield enforcement that goes beyond the shared design principles.

> **Read first:** `../../PRINCIPLES.md` (the four-tier shared principles).
> This file assumes you've already absorbed YAGNI / KISS / DRY / SRP / SOLID / layering / cohesion / coupling / etc., and only adds `evolve`-specific gates on top.

---

## Gate E1 — 200-LOC function hard cap

`evolve` enforces this absolutely (stricter than the shared "≤ 60 LOC soft target"):

- **Hard cap:** no single function > 200 body LOC (excluding signature, blanks, comments).
- **Trigger when violated:** the step is **not done**. Split before marking done.

### Standard splits (in order of preference)

1. **Validation prelude → `validateXRequest`** — one helper for input checks.
2. **Per-branch logic in a switch → one helper per branch.**
3. **Setup/teardown → fixture or table-driven case** (for tests).
4. **State machine step → table + dispatcher.**
5. **Long sequence of orchestration → named verb-phase functions called in order.**

### When > 200 LOC is genuinely necessary

Some real cases (a generated parser, a giant exhaustive switch over a closed enum, a hand-written codec).

- If the code is **generated**, it doesn't count — leave a `// generated by X` marker.
- If hand-written and unavoidable, **raise it as a refactor proposal** (`RP-NN: long function in <file>`) rather than silently exceeding the cap. The user can approve a documented exception.

## Gate E2 — Match existing conventions (consistency over local optimum)

New code must blend in with what `arch_scan/conventions.md` already documented:

- Same formatter (gofmt / rustfmt / prettier / black / ruff format) — run after every step.
- Same linter — zero warnings on new code, or accept-list with reason.
- Same error wrapping idiom (`fmt.Errorf("op: %w", err)` / `anyhow::Context` / etc.).
- Same logging library + structured field names. Don't introduce a second logger.
- Same DI pattern. Don't sneak in globals if the codebase uses constructor injection.
- Same test framework + assertion style. Don't add a second one.
- Same migration tool, same config loader, same HTTP/RPC framework.

If you genuinely think a convention is wrong → **refactor proposal**, never a quiet second style.

This is the operational form of "Consistency > local optimum" (PRINCIPLES.md Tier 4).

## Gate E3 — Refactor-Proposal Rubric

Phase 3 (and Phase 4 mid-implementation) raise existing code as a refactor proposal **only if** at least one of these is true:

1. **Blocks the new feature** — without the change, new code would have to duplicate logic, cross layers improperly, or weaken its tests.
2. **Hides a real bug** — silent error, wrong type assumption, race the new feature would amplify.
3. **Violates the project's own conventions** measurably (against `arch_scan/conventions.md`). Pure stylistic preference is **not** enough.
4. **Function exceeds the 200-LOC cap and the new feature must edit it** — touching it without splitting locks the violation in.

### Not valid reasons

- "I would have written it differently."
- "This module could be more elegant."
- "Add a layer of abstraction in case we need it later." (YAGNI — see PRINCIPLES.md Tier 1.)
- "Boy Scout Rule" used silently. (See PRINCIPLES.md Tier 3 — gated for `evolve`.)

### Required fields per proposal

- **ID:** `RP-NN` (zero-padded, sequential).
- **One-line title.**
- **Why blocking** (1–2 sentences, concrete).
- **Risk:** low / medium / high — based on test coverage of affected code.
- **Files touched** (full paths).
- **Cost in steps** (not hours).
- **Test situation:** covered / partially covered / needs new tests.
- **Definition of done:** measurable (e.g. "function ≤ 200 LOC, no behavior change, existing tests still pass").

## Gate E4 — Tests live with code (no test-debt deferral)

- Every Phase 4 step writes its production code AND its attached tests in the same step.
- Phase 5 is the regression loop, **not** a "now write missing tests" phase.
- A step is not done until its attached unit + interface tests are passing.

---

## Gate E5 — Phase 5 bug-zero hygiene

In the regression loop:

- A "fix" must be either a code change or a (test + spec) change with rationale logged in `coverage/failures.md`.
- "Rerun and it passed" is **not** a fix — flaky tests must be made deterministic, not retried.
- A test cannot be deleted unless the underlying spec in Phase 2 is also updated. Log the deletion + reason.
- A test cannot be silently rewritten to pass.
- After **3 rounds** with the same module showing failures, the skill writes a `# structural` note in `coverage/failures.md` and **considers raising a refactor proposal** (returns to the Phase 4 gate semantics).

## "Done" checklist for one Phase 4 step

Apply at the end of every step, before flipping status to `done`.

```
[ ] Implements only what the step card lists (no scope creep)
[ ] Compiles / type-checks
[ ] Project linter clean (or accept-listed with reason)
[ ] Project formatter applied
[ ] Attached unit tests written & passing
[ ] Attached interface tests written & passing (if applicable)
[ ] Gate E1: no function > 200 LOC
[ ] Gate E2: matches arch_scan/conventions.md (style, errors, logging, DI)
[ ] PRINCIPLES Tier 1 sanity: no YAGNI / KISS / DRY / SRP violation introduced
[ ] No silent refactor of pre-existing code (any change to existing files justified by step card OR an approved RP)
[ ] Step status flipped to `done`; state.json current_step++
[ ] Stop and report: files written/modified, LOC, tests run, pass/fail
```

## Quick decision flow during implementation

```
About to write a function?
  ├── Will body exceed ~60 LOC?         → plan splits up front
  ├── Will body exceed 200 LOC?         → MUST split or raise RP
  └── Reuses logic that already exists? → grep first, don't re-add (DRY rule of three)

About to touch a pre-existing file?
  ├── Step card lists it explicitly?    → OK
  ├── Approved RP covers this change?   → OK
  └── Otherwise                          → STOP. Either skip or raise RP.

About to introduce a new abstraction (interface / generic / plugin)?
  ├── Real second consumer exists?      → defer (Rule of Three)
  ├── Real third consumer exists?       → OK
  └── Otherwise                          → use concrete type, add abstraction later.

About to "clean up while I'm here"?
  ├── In approved RP?                    → OK
  └── Otherwise                          → STOP. Raise RP, don't silently change.
```
